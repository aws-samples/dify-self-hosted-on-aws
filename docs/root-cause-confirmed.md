# Dify パフォーマンス問題 — 調査結果サマリー（確定版）

> **ステータス：** 根本原因確定 ✅  
> **調査日：** 2026年4月3日  
> **調査方法：** AWS ECS Container Insights メトリクス（リアルタイム計測）

---

## 1. 計測結果（グラフで証明済み）

| メトリクス | 現在の値 | 正常範囲 | 判定 |
|---|---|---|---|
| CPU 使用率 | **105%** | 70%以下 | 🔴 限界超過 |
| メモリ使用率 | **99.76%** | 80%以下 | 🔴 限界超過 |
| タスク状態 | **クラッシュ → 再起動を繰り返し** | 常時1台稼働 | 🔴 不安定 |

---

## 2. なぜこうなっているか

現在の API タスクは **1 vCPU / 2 GB RAM** しか割り当てられていないにもかかわらず、以下の複数プロセスが同時に動作しています：

```
┌─────────────── 1 vCPU / 2 GB しかない ───────────────┐
│  ① API サーバー    （メインの処理）                    │
│  ② PluginDaemon   （プラグイン管理）                   │  ← 全プロセスが
│  ③ Sandbox        （コード実行）                       │     1 vCPU を奪い合い
└──────────────────────────────────────────────────────┘
         ↑ 物理的に処理しきれていない
```

---

## 3. 症状との直接的な関係

| 症状 | 原因 |
|---|---|
| UI がフリーズ・ローディングスピナーが止まらない | CPU が 100% 超で API レスポンスがタイムアウト |
| 保存した内容が消える | リクエストが処理しきれず失敗 |
| しばらくすると自然に回復する | タスクがクラッシュ後に再起動しているだけ |

---

## 4. ワークアラウンドについて

**有効なワークアラウンドはありません。**

- CPU が常時 100% 超のため、特定のコードやバグの問題ではない
- アイドル状態でもリソースが限界に達している
- プラグイン無効化・ユーザー制限などは一時的な緩和にしかならない

---

## 5. 推奨する解決策

**API タスクのリソースを増強するのみ。コード変更は不要。**

| 項目 | 現在 | 変更後 |
|---|---|---|
| CPU | 1 vCPU | **2 vCPU** |
| メモリ | 2 GB | **4 GB** |
| Redis | cache.t4g.micro（0.5 GB） | **cache.t4g.small（1.4 GB）** |

**追加コスト：約 $20/月（約3,000円/月）**

---

## 6. なぜ $20/月 が正当化されるか

現状のままでは：

- ユーザーが保存するたびにデータが消えるリスクが続く
- タスクが 1〜2 時間ごとにクラッシュし続ける
- 問題が発生するたびに調査・対応工数が発生する

> **$20/月 vs 毎月の障害対応工数、どちらがコスト高かは明白です。**

---
